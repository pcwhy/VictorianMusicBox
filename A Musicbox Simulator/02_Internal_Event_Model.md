# 第二章 内部事件表：程序真正反复操作的到底是什么

## 这一章要解决的问题

很多人第一次接触这类程序时，会下意识地以为：既然输入已经被配成了完整音符，后面系统大概就一直围着“音符对象”“和弦对象”或者某种更乐谱化的树形结构工作。我们分析的软件 不是这样。它真正反复加工的，是一张统一的内部事件表。后面的转调、同拍加厚、颤音、高频滚动装饰、机械过滤和导出分叉，虽然听起来像在处理完全不同的问题，实际却都在围绕同一套中间表示动手。

这一章如果讲不清楚，后面的章节读起来就会像在不停换对象：前面好像在讲乐理，中间忽然改讲机械，后面又变成导出文件格式，读者会自然怀疑“这些到底是不是同一个系统”。这也是为什么第二章必须尽早把这张内部事件表摆出来。它不是某个实现细节，而是整条主链真正共享的工作底板。

## 什么叫“内部事件表”

这里说的“内部事件表”，不是数据库术语，也不是某种抽象得很高的中间语言。它更像一张流水线工单表：每一行代表一个后续阶段还能继续加工的事件，这一行至少要告诉系统它发生在什么时候、它属于什么类型、它携带什么主要值，之后的阶段再决定要不要继续追加新行、改写现有行、把局部顺序重新排一遍，或者干脆过滤掉一部分结果。

这和“乐谱树”的思路差别很大。乐谱树适合表达理想化的音乐结构，比如一首曲子有几段、每段有几个和弦、每个和弦里有哪些音。但 我们分析的软件 真正难的地方，恰恰不是怎样优雅表达“乐谱本体”，而是怎样保住那些在现实加工链里很容易丢掉、却会显著改变结果的局部关系：同拍但来自不同轨道的事件如何混在一起，重复音后面怎样继续参与 ornament，新长出来的装饰音怎样和旧音一起排队，导出时谁先 note-on 谁后 note-off，以及某个事件最后是不是会在机械层面被裁掉。要承载这些东西，一张可以继续追加、改写和重排的事件表，反而比一棵漂亮的对象树更贴近系统本来的业务需要。

## `GridRow` 到底是什么

当前主线代码里，这张内部表的基本单元叫 `GridRow`。定义非常直接：

```python
@dataclass
class GridRow:
    time: int
    type_code: str
    value: int
    track: int
    channel: int
```

这五个字段看上去很简短，但它们已经足够支撑后面的大部分业务判断。`time` 说明这个事件发生在第几个 `tick`，`type_code` 说明它属于哪一类事件，`value` 在当前主链里大多数时候承载的是音高值，`track` 记住它最初来自哪条轨道，`channel` 记住它最初属于哪一路 MIDI 通道。换句话说，一行 `GridRow` 同时保留了时间事实、事件类型、音乐内容以及导入来源。

最小例子可以直接写成：

```text
GridRow(time=192, type_code="3", value=64, track=1, channel=0)
```

如果把它翻成普通话，这行的意思就是：在第 `192` 个 `tick` 上，有一个普通发音事件，它的音高值是 `64`，它最初来自第 `1` 条轨道，属于第 `0` 号 MIDI 通道。如果读者愿意，也可以先粗略把它想成：“在 `192 ticks` 这个时间点，第 `1` 条轨道上的一个普通乐器音 `E4` 进入了内部流水线。”

这个例子的重要性，不在于让读者记住 `64 = E4`，而在于让人真正看到：系统后面操纵的不是“一个抽象旋律音”，而是一条带着时间、类型、轨道和通道来源的具体事件。

## 为什么 `type_code` 绝对不能省

很多第一次看这张表的人，都会本能地觉得 `type_code` 很像一种可以以后再推断出来的冗余字段。对这套系统来说，它不是。因为后面的阶段并不只问“这里有个什么音”，它还不断问这条事件现在处在什么业务身份里：它只是普通进入流水线的发音事件吗？它是不是已经被标记成足够长、允许继续做 ornament 的长音？它是不是某个后续阶段新增出来的装饰事件？

当前最常见的两类可以先记成这样：`"3"` 通常表示普通进入流水线的发音事件，`"4..."` 则表示带时值信息的长音事件。你不必在这一章就背全所有类型码，但必须先接受一个事实：后面的很多判断不是只看音高和时间，还要看“这条东西在系统眼里现在算什么”。这也是为什么同样一个 `64`，有时只是普通发音，有时却已经被后面的阶段看成“可以继续加工的长音原料”。

## 为什么这里不用“对象树”或“乐理树结构”

如果从现代软件设计出发，很容易先想到另一条路线：先建立一首曲子的层级结构，里面有若干段落，每段有若干和弦，每个和弦里又有若干音，这些音再挂更多属性。这样的结构当然可以做，也确实很适合很多乐谱软件。但它天然会鼓励一种看法：系统面对的是“理想乐谱”，后面的大部分行为只是往这份理想乐谱上继续加信息。

我们分析的软件 的问题恰恰更像相反的方向。它面对的不是一份完全干净的和弦树，而是一串会不断被重新组织的事件流。对它来说，真正敏感的是局部前后关系，而不是高层乐理标签。两份输入就算最后包含同样的音高集合，只要局部追加顺序不同、同拍事件在内部表里的邻接关系不同，后面的 family 结构和 ornament 结果就可能直接改观。换句话说，这里真正需要保住的是“事件如何进入系统、如何在局部被重排、如何和后来长出的新事件混在一起”，而不是先急着把它们升格成一棵优雅的乐理对象树。

因此，这一章的结论不是“对象树一定错”，而是“在这个系统的主链里，内部事件表是更接近业务现实的第一等公民”。如果一开始就把它藏掉，后面很多看起来诡异但真实存在的结果就会重新变回黑箱。

## 第一次排序到底在改什么

导入阶段先建立 `base grid` 之后，程序不会永远保持最初追加进去的顺序。它会很早做一次按时间的重排。这就是第二章必须讲清楚的“第一次排序”。

如果要把这一步写成一句正式定义，可以这样说：第一次排序是程序第一次对内部事件表做“按时间比较、整行交换、结果继续流向后续 ornament 阶段”的重排。它的作用不是把表弄得更整齐好看，而是把原本带着逐轨追加痕迹的原料表，转成后续阶段更容易消费的局部时间结构。

当前静态与 replay 证据都表明，这一步的比较规则非常克制：它主要按 `time` 比较，交换的是整行，而且它不是一个稳定排序。也就是说，系统此时关心的不是“做一个现代意义上最漂亮的排序实现”，而是把局部时间相近的事件重新聚到一起，同时保住那种会继续影响后续行为的整行关联。

最小例子最能说明它到底在改什么。假设导入阶段先逐轨追加出了下面四行：

```text
1. GridRow(time=240, type_code="3", value=64, track=1, channel=0)
2. GridRow(time=120, type_code="3", value=52, track=1, channel=0)
3. GridRow(time=120, type_code="3", value=59, track=2, channel=0)
4. GridRow(time=240, type_code="3", value=67, track=2, channel=0)
```

从“逐轨追加”的角度看，这个顺序并不奇怪，因为程序确实可能先把第 `1` 条轨道扫完，再去扫第 `2` 条轨道。但从后续 ornament 阶段的角度看，这个顺序并不理想，因为同一时间的材料被轨道读取顺序打散了。第一次排序之后，读者更应该期待看到这样的结果：

```text
1. GridRow(time=120, type_code="3", value=52, track=1, channel=0)
2. GridRow(time=120, type_code="3", value=59, track=2, channel=0)
3. GridRow(time=240, type_code="3", value=64, track=1, channel=0)
4. GridRow(time=240, type_code="3", value=67, track=2, channel=0)
```

注意，这里改掉的不是音高内容，而是局部邻接关系。也正是这种关系，决定了后面某个阶段眼里“眼前这一小团材料”究竟长什么样。

## 为什么 `base grid` 建出来以后还一定要再排一次

这一步如果不讲透，读者会自然产生一个疑问：既然每行已经带 `time`，为什么不直接往后走？答案在于，导入阶段虽然已经知道每个音发生在什么时间，但 `base grid` 并不是按“全局时间顺序”建出来的，它是按“逐轨读取、逐条追加”的顺序建出来的。

当前实现里，`build_base_grid(...)` 的外层循环大致是：

```text
for track_events in midi.tracks:
    for event in track_events:
        ...
        rows.append(...)
```

这说明 `base grid` 的初始行顺序首先受两件事支配：它属于哪条轨道，以及这条轨道内部事件原本怎样往前走。换句话说，它先保住的是“轨道主导的原始追加顺序”，而不是“全局时间已经彻底洗平后的顺序”。

导入阶段故意保住这些事实，是因为它们对还原原程序很重要。系统需要记住事件来自哪条轨道，它们是按什么追加顺序进入内部表的，以及同拍事件最初怎样前后排开。但这些事实和“后面的 ornament 阶段最希望看到的局部结构”并不是同一回事。后面的阶段更关心的是：哪些事件在时间上属于同一小团，这些事件重新聚拢后当前 family 长什么样，以及哪些长音、同拍音和新增材料应该先被看见。

所以 `base grid` 建出来以后再排一次，不是在修正导入错误，而是在完成一个正式转换：把“保留原始进入系统顺序的原料表”变成“更适合后续 ornament 阶段消费的局部时间结构”。

如果跳过这一步，系统当然也能继续跑，但后面会出现两个非常真实的问题。第一，同样的音高集合会因为局部前后顺序不同，长出不一样的 family。第二，ornament 阶段看到的候选材料会偏向“谁先被追加进来”，而不是“谁在时间上更应该先被归到一起”。因此第一次排序不是可有可无的清洁动作，而是主链第一次正式把“导入事实”转成“可加工局部材料”的地方。

## 为什么“音高总数一样，但结果还是不同”

这是重构时最容易产生错觉的地方。很多人看到两份结果，会先做一种很自然的比较：总共有同样多的音，几个关键时间点上也都能对上，于是就觉得两边应该已经等价了。对这个系统来说，这样的比较还不够细。

原因在于，后面的阶段并不是只看“最后剩下了哪些音”，它还在看这些音在内部事件表里是怎样彼此挨着的，谁先被扫到，谁和谁会暂时被看成同一小团局部材料，以及同拍事件在局部有没有重新排过顺序。只要这些关系变了，后面长出来的 family 结构和 ornament 结果就可能一起变掉。

所以这里真正该比较的，不只是“音高集合有没有一样”，而是“这些音在内部表里怎样组成后续阶段要消费的局部结构”。如果这一层不一样，那么即使总音数一样、主要时间点也差不多，后面仍然可能走出完全不同的结果。

## 为什么内部事件表是主链核心，而不是实现细节

现在回头看，内部事件表之所以关键，不是因为它“方便写代码”，而是因为它统一承载了几类原本很容易被拆散的问题：音乐材料怎样被组织，装饰音怎样被追加，机械过滤怎样决定保留或丢弃，以及 `.nts`、processed MIDI、动画怎样从同一来源分叉出去。只要这一层的语义被弄乱，后面每一层都会变得难以解释。

从这个角度看，这张表不是某个临时中间结构，而是整个主链真正共享的加工底板。前面的导入阶段把事实放进来，后面的 ornament 和机械阶段在上面继续改写，最后输出层又从这里分叉出去。你可以在工程实现上换数据结构名字，甚至换语言，但很难在业务上绕开“必须有一套统一中间表示”这个事实。

## 对重构最重要的启示

这章最后真正想留下的结论只有三个。第一，不要只保留“干净音符对象”，还要保留一张正式的内部事件表。第二，这张事件表必须支持统一追加和局部重排。第三，第一次排序必须被建成明确的正式阶段，不能悄悄藏进某个匿名工具函数里。只有这样，后面的 ornament、机械约束和输出分叉才会继续落在同一条可解释的主链上。
