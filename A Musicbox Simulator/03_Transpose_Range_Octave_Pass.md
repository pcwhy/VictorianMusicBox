# 第三章 音区整理与同拍加厚：`Transpose / Range / Octave pass`

## 这一章在主链里负责什么

这一阶段最容易被误解成一句很简单的话：“把所有音统一升高或降低一点。”这并不对。它真正做的是两件连在一起、但又不能混成一件事的工作。第一件事是根据整首歌的音域，决定整体往哪个工作音区搬；第二件事则是在这个整体搬移完成以后，对一部分转调后的音做同拍加厚。因此它既是音区整理阶段，也是后面 ornament 阶段的材料准备阶段。更顺的理解方式，是先把它看成“整首歌先搬家”，再把它看成“搬完以后，一部分音被加厚”。两者是一前一后的两步，但不是一回事。

先把这三个词拆开。这里的 `Transpose` 指整体移调，它关心的不是“乐理上转了什么调”，而是整首歌应该整体往上或往下搬多少，才更适合这台八音盒系统的工作区间。`Range` 指的是音域整理，它不是单独的一步动作，而是决定整体移调时的目标：让整首歌尽量落进机器更合适的音区。`Octave pass` 则是同拍加厚，它不是分解和弦，不是把一个和弦拆成前后滚动的音序列，而是在同一个开始时间点再追加一层或两层相关音，让这一拍更厚、更像八音盒的质感。

这一步为什么必须先看整首歌，而不是看到一个音就立刻决定怎么移调，背后的道理其实很朴素。程序会先看整首歌的最低音和最高音，再决定整体偏移量。当前实现里，核心输入就是 `min_note` 和 `max_note`，也就是整首歌的最低音和最高音。先看整体再决定偏移，意味着系统相信：如果每个音各自想办法“适配机器”，结果会非常不稳定；只有先决定全局工作区，再处理单个音，结果才会稳得多。

## 整体移调怎样决定目标音区

先只说第一部分，也就是转调与音域整理。它要回答的核心问题只有一个：整首歌应该整体往上或往下搬多少，才更适合这台机器。当前主线代码在 `compute_transpose_pass_defaults(...)` 里把这一层的默认规则写得很明确。先定义几个变量：`min_note` 是整首歌最低音，`max_note` 是整首歌最高音，`base_mid = 60` 是计算工作上限时使用的基础中点，`low_target = 100` 是希望最高音大致靠近的位置，`high_limit = 94` 是当前代码里保留下来的历史常量，而 `avg = (max_note + min_note) // 2` 则表示整首歌音域中点。

程序先算一个 `upper_bound`。它不是最终输出音高，而是后面判断“哪些音允许做同拍加厚”的工作上限：

```text
upper_bound = 60
if avg > upper_bound:
    upper_bound = avg
if upper_bound > 72:
    upper_bound = 72
```

这一步的意思是，系统先根据整首歌的中间音区估一个“比较适合加厚的上边界”，但这个边界不会无限升高，最多只到 `72`。接着它再算整体移调量 `offset`：

```text
if low_target + min_note - max_note < 31 and low_target <= max_note:
    offset = 0
else:
    offset = low_target - max_note
    if offset <= 5:
        if offset > 0:
            offset = 0
    else:
        offset -= 5
```

如果把这段伪代码按业务语义直接翻译出来，`offset` 就是最终要给整首歌所有音统一加上的半音位移。当前规则首先会判断：如果原曲音域已经够宽，而且最高音已经不低，那就不要再整体上移，直接把 `offset` 设成 `0`。否则就先尝试把最高音推向 `low_target`。但即使如此，系统也不会对很小的正向上移过于敏感：如果只是轻微上移，它会干脆把这点位移压掉；如果确实需要明显上移，则会保留大方向，但再减去一点保守余量。

最后，程序还会让 `upper_bound` 随着 `offset` 一起平移，再把它夹回 `60..72`：

```text
upper_bound += offset
if upper_bound > 72:
    upper_bound = 72
if upper_bound < 60:
    upper_bound = 60
```

如果不想先死记公式，也可以先抓它的业务意图。第一，它倾向把整首歌往较高、较明亮的工作区搬；第二，它不愿意为了一点点小差距就做很敏感的小移调；第三，它顺手给后面的 `Octave pass` 准备了一个相对保守的工作区间。所以这一步追求的不是“数学上最完美地居中”，而是先让整首歌落进一个更像八音盒的工作音区，再把“哪些音允许进一步加厚”这个问题留给下一步。

## 同拍加厚怎样在当前这一拍扩音

到了第二部分，也就是 `Octave pass`，程序已经先做完整体转调。现在它开始问第二个问题：哪些转调后的音值得在同一拍再加几层？这就是为什么前面不能一开始就把“转调”和“同拍加厚”搅在一起。它们在业务上是一前一后的两步。

在当前实现里，程序会检查每一个转调后的音 `new_note`。只有当它满足 `48 <= new_note <= upper_bound` 时，才会触发同拍加厚。默认规则在 `expand_transpose_default(...)` 里写成：

```text
down12 = new_note - 12
extras.append(down12)

if 36 <= down12 <= 48:
    extras.append(down12 + 24)

if down12 >= 48:
    extras.append(down12 - 12)
```

业务上可以这样读：先无条件补一个低 12 半音的音；如果这个低 12 半音后的值还落在 `36..48`，再补一个再高 24 半音的音；如果这个低 12 半音后的值已经不低，再补一个更低 12 半音的音。这些新增音与原音有同一个开始时间和结束时间，所以它们是同拍扩张，而不是滚动装饰。

举一个具体例子会更容易懂。假设某个音经过整体移调后变成 `69`。因为 `69` 落在 `48..upper_bound` 里，所以它会进入 `Octave pass`。先算 `down12 = 69 - 12 = 57`，于是一定会追加 `57`。接着看条件：`57` 不在 `36..48` 里，所以不会追加 `57 + 24`；但 `57 >= 48`，因此还会再追加 `57 - 12 = 45`。于是最终这一拍会同时出现 `69, 57, 45`。这就是为什么它听起来像“厚了一层”，却又不是分解和弦。

## 为什么这一层会直接影响后面的 ornament

这一层最重要的下游作用，不是“这一拍当下更满”，而是它改变了后面看到的同拍材料数量和形状。如果输入本来只有一条很瘦的单音线，后面的 ornament 阶段可能根本没有足够材料可用；而当这一层把某些位置扩成更厚的同拍簇之后，后面就更可能出现更大的 candidate 集、更像八音盒的高频装饰，以及更丰富的局部 family。所以它本身虽然不是 ornament，却会直接决定 ornament 是否有机会长出来。

也正因为如此，用户常会产生另一种误判：明明已经打开了这一层，为什么听起来还是没有立刻更华丽？常见原因有三类。第一，它确实只做了整体音区整理，没有在当前材料上触发很多同拍加厚；第二，它已经把材料加厚了，但后面的 ornament gate 没有通过；第三，前面两步都通过了，后面又被机械可播放性过滤掉了一部分。所以“没有明显更华丽”并不自动等于这一层没有工作。

## 这一层在重构里应当保留成什么

走到这里，这一层的业务边界也就清楚了。它不是一个模糊的“音符美化器”，而是先做整体音区整理，再做条件性的同拍加厚。后面如果要继续解释为什么某个位置会长出 ornament、为什么另一个位置却没有，首先就要回头看这里有没有把整首歌搬进合适的工作音区，又有没有准备出足够厚的同拍材料。
