# 第八章 输出分叉：为什么 `.nts`、processed MIDI、WAV 和动画不会完全一样

## 为什么这几种输出本来就不该完全一样

很多人第一次接触这套系统时，会很自然地带着一种现代软件直觉：输入既然只有一份 MIDI，那么后面出现的 `.nts`、processed MIDI、WAV 和动画，大概只是同一结果的不同文件格式。对这套系统来说，这种直觉会马上出错。它从一开始就不是把“同一份最终结果”分别存成四种外壳，而是允许声音侧、机械事件侧和动画侧各自保留不完全相同的结果。这不是实现污染，也不是几条输出链后来互相漂移，而是系统设计本身的业务分工。

要理解这一章，最重要的不是先死记四种输出的名字，而是先接受一个前提：这套程序并不把“最终声音”当成唯一真相。它同时试图描述几件彼此相关、却不完全相同的事情。第一件事是机械上什么时候触发了什么内部音值；第二件事是这些内部结果怎样重新写回一份可供软音源继续合成的 MIDI；第三件事是最后真正听到的声音是什么；第四件事则是如果把整台 Regina 的机械动作画出来，哪些触发应该被表现出来。只要目标不同，输出就不可能完全一样。

## `.nts` 为什么不是最终声音文件

在这四条输出线里，`.nts`` 最容易被误会。很多人第一次看到它，会下意识以为它已经是盘片孔位表，或者至少已经是“最终机械动作”的完整记录。其实它更接近一份机械事件时间表。当前证据已经足够把它写成一个明确的数据结构：

```text
record size = 5 bytes

struct NTSRecord {
    int32 event_time_ms_scale;
    uint8 note_value;
}
```

这意味着 `.nts` 每条记录只保留两件事：在什么时间触发，触发哪个内部音值。它没有表头，不写力度，不写时值，不写轨道，也不写通道。它甚至不直接写 Regina 的机械槽位号，而是继续使用系统内部的音值表示。因此，`.nts` 不是“最终声音的另一种编码”，也不是“动画文件”；它是一种机械事件层的正式中间格式。系统之所以需要它，是因为它想把“声音怎么合成出来”和“机械侧怎样被触发”拆开保存。

## processed MIDI 为什么不是中间表的直接镜像

processed MIDI 负责的却是另一件事。它仍然是一份 MIDI 乐谱，但已经不是原始输入 MIDI，也不是简单把中间结果逐音抄回去的导出。它必须经过一层 writer 逻辑，才能重新变成标准 MIDI 轨道事件流。这里的 writer，不该被理解成“最后负责写文件的小函数”，而应该理解成“把内部结果重新整理成标准 MIDI 事件的写出层”。正因为有这一层，系统才需要重新决定 tempo 事件放在哪条轨道上，同 tick 的 note-on 和 note-off 如何交错，同音重触发怎样排序，以及某些 note-off 是否需要省略。processed MIDI 因此不是中间结构的镜像副本，而是声音合成路径专用的整理版本。

这也解释了为什么 processed MIDI 和 `.nts` 明明共享同一份内部加工结果，最后看起来却不会完全一样。`.nts` 只关心“什么时候触发哪个内部音值”，而 processed MIDI 还要关心标准 MIDI 写出规则、同音重触发、note-on 与 note-off 的相对顺序，以及更后面的机械过滤结果。前者更偏机械事件表达，后者更偏声音合成中间层。它们当然有共同来源，但设计目标从一开始就不同。

## 最终 WAV 是怎样从 processed MIDI 长出来的

WAV 这一层离机械事件又更远了一步。最终 WAV 不是系统内部直接算出来的，也不是从 `.nts` 反推出来的，它来自 processed MIDI 再经过外部软音源 TiMidity 的合成。当前逆向和文档证据已经能把这个流程说得比较清楚：程序先写出 `jjpfile.mid`，再启动 `timidity.exe`，把这份 processed MIDI 交给 TiMidity 合成为 `jjpfile.wav`，最后还会把一段机械 rumble 底噪拼到前面。因此最终 WAV 不是“纯 MIDI 回放”，而是“TiMidity 合成出的音乐主体，再加一层机械启动和运转的声音感”。

这里还牵出两个读者常会问的问题。第一个问题是 `regina15.pat` 到底是什么。当前仓库里没有这个文件本体，因此正文不能假装已经把它的内部内容全看到了；但从调用位置、文件名和 TiMidity 的用法，已经足以确认它在业务上的角色：它不是输出文件，而是供 TiMidity 使用的一套 Regina 风格音色或 patch 资源。它决定的不是“什么时候发音”，而是“发出来的声音质感像不像这台八音盒”。第二个问题则是为什么命令行参数看起来没有完全写死。这里最诚实的结论是：我们已经确认程序会调用 `timidity.exe`，会把 `regina15.pat` 和输入 MIDI 一起传进去，也已经在反编译里看到了诸如 `-OwM`、`-f -s 44100 -a -p48 -o`、`-f -s 11025 -a -p48 -o` 和 `\jjpfile.wav -P` 这类参数组；但由于反编译结果没有把最终命令行字符串以完全无歧义的形式直接拼出来，正文最合适的写法不是假装已经还原出唯一完整命令，而是把已经确认的参数组及其控制目标写清楚。对重构文档来说，这样更诚实，也更有用。

## 为什么动画路径还会再分出去一次

动画这条线又更不一样。很多读者会本能地觉得，既然最终听到的是一段音乐，那动画大概就是“把这段音乐画出来”。这在这套系统里同样不成立。动画的首要目标不是忠实表现你耳朵里听见的所有细节，而是尽量表现一张真实 Regina 盘片在机械上会怎样工作。这会造成一个关键分叉：某些音在声音路径里还能被听到，但在动画路径里不应该出现，因为真实机械结构根本打不出来。换句话说，动画服从的是“机械上如何运动”的约束，而不是“最终音频里听到了多少东西”的约束。

理解了这几条分叉，再回头看“为什么四条输出不一样”，就会发现它其实不是异常，而是自然后果。`.nts` 只保留机械事件时刻和内部音值，processed MIDI 要重新遵守标准 MIDI 写出规则，WAV 还要通过 TiMidity 合成并拼上 rumble，动画则要优先服从机械动作可解释性。只要目标不同，四条输出线就不应该被强行统一成同一份结果的四种文件格式。

## 哪些 MIDI 信息会在这些分叉里被弱化或丢弃

这一点在“哪些 MIDI 信息会一路保留下来，哪些会被弱化或丢弃”时表现得尤其明显。从现代 MIDI 用户的角度看，这套系统确实会丢掉很多原本看似理所当然的表达层。导入阶段就会丢掉除 tempo 以外的大部分 meta 事件、SysEx、Program Change、Pitch Bend、Polyphonic Aftertouch、Channel Pressure、大部分控制器变化以及 `note off` 的力度。还有一些信息虽然会被读进来，却不会成为 ornament 决策的主轴，最典型的就是 `note on` 的力度：它会被保留一段时间，也可能继续被写回 processed MIDI，但 ornament 是否生成、生成哪一类 ornament，主要并不靠它决定。等走到输出分叉时，丢失会变得更进一步：`.nts` 会直接丢掉力度、时值、轨道和通道；processed MIDI 虽然还保留力度，但 note-off 会被重写成标准写出形式；某些业务模式还会在进入输出链之前主动删掉鼓通道。这里尤其需要强调的是，“鼓点最后没出现在输出里”和“导入阶段根本没把鼓读进来”不是一回事。前者是后续业务链的主动过滤，后者才是导入丢失。

## 这一章在重构里真正要保住什么

于是我们终于可以把这一章的核心结论说清楚：`.nts` 不是最终声音，processed MIDI 不是动画数据，WAV 也不是系统内部唯一的真相，而动画更不是“可视化版 WAV”。它们都只是同一内部加工结果在不同业务目标下的不同投影。对重构来说，正确做法不是强行发明一个“万能最终结果对象”，让这四条线都去服从它，而是先明确它们究竟共享哪一层中间结果，再明确它们从哪一步开始分叉，以及每条分叉各自服务什么目标。只有这样，当它们彼此不完全一致时，重构者才能判断那是设计本来就需要的差异，还是实现真的出了错。
