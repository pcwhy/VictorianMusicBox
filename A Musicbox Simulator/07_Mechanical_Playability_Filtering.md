# 第七章 机械可播放性阶段：槽位映射、最小间隔和“机械不可能音”

## 这一章要解决的问题

这一章回答的是一个非常现实的问题：一段音乐在乐理上成立，不代表 Regina 这样的实体机械系统真的来得及把每个音都打出来。程序必须把“音乐上存在的音”再翻译成“机械上是否能触发的音”，这一步就是 mechanical playability filtering。

## 先定义“机械不可能音”

一个音是 `mechanically impossible`，不是说它乐理上错了，而是说：

它在当前时间点和当前机械位条件下，对真实机器来说来得太快，或者同一个机械位置还没来得及恢复，因此实体机器实际上打不出来。

所以“机械不可能”是机械约束概念，不是音乐正确性概念。

## 这一步不是先问“删不删音”，而是先问“它落在哪个机械位”

程序不会直接对一个 MIDI 音高说“这个音保留，那个音删除”。它真正先做的，是把音高映射到一个或多个候选机械位，再对当前这一条机械事件在这些候选位之间选一个要尝试使用的位，最后检查这个位与上一次触发之间的间隔够不够。所以这一步的核心不是一个粗暴的 `if playable else drop`，而是一套“映射、选位、检查最小间隔”的流程。

## 这里说的“当前这一条机械事件”，时间尺度有多宽

这一点必须说得很硬，因为读者很容易把它误会成：

一个小节、一拍，或者某个更宽的局部时间窗。

当前实现里，这里的单位其实非常窄：

`filter_notes_for_audio(...)` 会先把每个音变成一条二元记录，也就是 `event_time_ms` 和 `note_value`；然后 `assign_mechanical_slots(...)` 再按这些记录逐条处理。

所以这里说的“当前这一条机械事件”，实际就是：

某一个具体的 `event_time_ms` 上的一次单独触发尝试。

更直白地说：

程序每处理到一条记录，就为这一条记录选一个候选机械位，然后立即检查这个位离它上一次触发是否太近。

因此这里的时间尺度不是“一个窗口”，而是：

一条单独事件、一个具体毫秒时间戳，以及一次单独的机械触发判定。

## 为什么同一个音高会有多个候选机械位

如果同一个音高永远只有一个固定机械位，那么连续重复时会非常容易撞上最小间隔限制。  
给它两个或三个候选位，程序就能在这些位之间轮换。

因此一个音高有多个候选位，不是为了“高级”，而是为了更接近真实机械系统的可播放策略。

最小例子可以这样想：假设某个音高 `64` 允许映射到两个候选机械位，分别是 `26` 和 `27`。如果上一次 `26` 号位刚在 `1000 ms` 被触发，而这次新事件出现在 `1300 ms`，同时 `26` 号位的最小 gap 恰好是 `350 ms`，那么这次就来不及再用 `26`。程序不会立刻删掉这个音，而是会先试着看：

`27` 号位是不是刚好空着，或者它离上一次触发已经足够久。

如果 `27` 可以用，这个音就还能保住。  
只有当两个候选位都来不及恢复时，这个音才会在机械层面失败。

## gap 为什么不是常数

这也是最容易让人误会的地方。  

很多人会直觉以为机械最小间隔应该是一个全局固定值。

不是。  
当前实现明确表明：

最小 gap 按机械位变化，而不是一个全局常数。

主线代码里对应的是 `build_slot_min_gaps()`，核心公式分两段：

### 对 `slot = 1..43`

```text
x = (slot - 1) * 0.08333 + 0.8
gap_ms = round(0.29 / (x * 2π) * 60000)
```

### 对 `slot = 44..76`

```text
x = slot * 0.08333 + 0.8
gap_ms = round(0.29 / (x * 2π) * 60000)
```

这意味着：

槽位不同，最小允许重复触发间隔不同；高位与低位的限制不完全一样；`43` 和 `44` 之间还有一个额外的几何断层。

业务上可以简单理解成：

这台机器不同位置的“恢复速度”不一样，程序把这种差异写成了每个槽位自己的最小 gap。

## 程序怎样决定一个音是否失败

当前主线代码的判定可以概括成：

```text
if last_time_of_slot != -1
   and current_time - last_time_of_slot < slot_min_gap[slot]:
       这个音本轮写成 slot = 0
else:
       这个音保留，并更新这个槽位的最近触发时间
```

这里的 `slot = 0` 不是一个真实机械位。  
它的业务含义是：

这一条音在机械可播放性层面失败了。

把上面那种流程完整写成一个最小事件例子，会更容易懂。假设当前这一条记录是：

```text
event_time_ms = 1300
note_value    = 64
candidate_slots = [26, 27]
```

再假设程序查到：

```text
last_time_of_slot[26] = 1000
last_time_of_slot[27] = 980
slot_min_gap[26]      = 350
slot_min_gap[27]      = 400
```

那么这次与两个候选位的间隔分别是：

```text
1300 - 1000 = 300 ms
1300 - 980  = 320 ms
```

这两个间隔都小于各自要求的最小 gap，于是两个位都不能用。程序最后就会把这一条记录写成：

```text
slot = 0
```

如果同样的例子里，`27` 号位上一次触发时间改成 `850 ms`，那么：

```text
1300 - 850 = 450 ms
```

这次就已经大于 `400 ms` 的最小 gap，于是程序还能把这一条记录分配到 `27`，这个音就不会被过滤掉。

## `filter_notes_for_audio(...)` 实际在做什么

主线里的 `filter_notes_for_audio(...)` 不是按“音高喜不喜欢”“时值够不够长”过滤，而是按机械槽位检查的结果过滤。

它的流程可以概括成这样：先把音符按开始时间等字段稳定排序，再把每个音变成 `.nts` 风格的二元记录，也就是 `event_time_ms` 和 `note_value`，然后调用 `assign_mechanical_slots(...)` 给每个事件分配机械位。如果某条事件最后得到 `slot = 0`，系统就把它从音频和 MIDI 路径里滤掉。所以它的真实业务含义，不是“按音乐美感删音”，而是只保留当前机械上还来得及真实触发的音。

## `Play Mechanically Impossible Notes` 改变的到底是什么

这个选项最容易和上一章的 `Mechanical Timing Effects` 混掉。  
它真正改的是：

是否绕过音频和 MIDI 路径上的机械不可播放过滤。

也就是说：

默认情况下，机械上打不出来的音会被从音频路径里裁掉；如果用户明确允许 `Play Mechanically Impossible Notes`，这些音可以继续保留在声音结果里。

但有一个很重要的边界：

动画侧仍然应该遵守实体机械限制。

所以即使用户允许“音频里听到机械不可能音”，动画也不该假装盘片真的把这些音打出来了。

## 为什么音频和动画会分叉

这一步正是音频与动画最容易分叉的地方。  
因为一个音可能出现三种情况：

它可能机械可播放，于是音频和动画都保留；也可能机械上来不及打，但用户允许在音频中保留，因此声音里还有、动画里没有；还可能机械上来不及打，而且音频路径也不保留，于是两边都消失。

因此“我听到了，但动画里没有”在这里并不一定是 bug，而可能是设计本来就允许的结果。

## 最容易看错的地方

这一层最容易被看错的第一点，是把最小 `gap` 当成一个常数，好像所有机械位只要统一遵守同一个恢复时间就够了。真实情况更细：`gap` 会随着槽位变化而变化，甚至 `1..43` 和 `44..76` 还分成两段计算。也就是说，系统并不是拿一把统一尺子去裁所有音，而是在问“这个具体机械位，上一次刚刚被拨过之后，还来不来得及再打一次”。

第二个常见误读，是把音高直接等同于“能不能打”。这一层真正做的事情不是“看到某个 MIDI 音高就立刻判死刑”，而是先把音高映射到一组候选机械位，再根据这些机械位各自最近一次被触发的历史，判断这一次是否还来得及。因此决定结果的不是一个抽象音高本身，而是它落到具体机械位之后，与这些机械位当前状态之间的关系。

第三个常见误读，则是把 `Play Mechanically Impossible Notes` 理解成连动画都会一起放宽。按照当前理解，这个选项主要影响的是音频和 MIDI 路径，让某些机械上来不及打的音在声音侧仍然可以被保留；动画却仍然更应服从真实机械限制。所以“我听到了，但动画里没有”在这里并不天然是 bug，而可能正是两条输出路径本来就被设计成不同目标的结果。

## 对重构最重要的启示

如果要现代化重构这一层，至少要保住四件事：先映射候选机械位，再判断可播放性；最小 gap 是按槽位变化的，而不是常数；机械失败应有一个明确的中间状态，例如 `slot = 0`；音频路径和动画路径在这里允许分叉。如果把这一层简化成“按音高粗暴删音”，系统固然会更简单，却会不再像原来的 Regina 机械模型。
