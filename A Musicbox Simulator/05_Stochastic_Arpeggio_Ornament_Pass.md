# 第五章 高频滚动装饰：为什么有时会很漂亮，有时却什么都没有

## 这一章真正想回答什么

很多人打开高频装饰选项之后，会自然期待一种很稳定的结果：某一拍里已经有几颗音，程序就把它们自动铺成一串漂亮的八音盒滚动装饰，而且每次效果都差不多。这套程序不是这样工作的。它不是一个“看到和弦就展开”的固定模板引擎，而更像先观察眼前这一小团材料够不够丰富；如果不够，就什么都不做；如果够，再决定这一次要不要真的继续；即使真的继续，最后长出来的样子也不止一种。所以这一章真正要回答的，不是“它会不会分解和弦”这么简单，而是这一步到底在看什么、为什么很多地方完全不会出装饰，以及真出装饰时大致会长成什么样。

## 这一步和传统分解和弦是什么关系

如果先不用内部名字去理解，这一步可以暂时被读成一句业务描述：程序从一个很短的局部时间范围里挑出一小团音，把它们滚成一串更密、更亮的高频装饰。它和上一章最大的区别在于，上一章更像“把一个长音切细”，而这一章更像“把一小团同时或近乎同时出现的音展开成一串滚动纹理”。如果一定要找一个现代音乐制作里接近的概念来类比，可以把它理解成一种受约束、带概率性、并且明显偏八音盒风格的短分解装饰，但它又不是现代 DAW 里那种完全可预测的固定 arpeggiator。

这里最好把一个古典乐理词也顺手讲清楚。`arpeggio` 在传统乐理里，通常指把原本可以同时按下的一组和声音，改成前后依次响起，而不是同一瞬间一起响。最容易懂的例子是：如果某一拍理论上可以同时响 `C / E / G` 三个音，那么 `arpeggio` 的直觉效果更像先响 `C`，再响 `E`，再响 `G`，让和弦从“一个厚块”变成“一小串滚动音”。这一章说的高频滚动装饰和这个概念是亲戚，但不是同一个东西。它不像古典乐谱里的标准分解和弦那样稳定、明确，也不是看到任何和弦都照模板展开。它更像：程序先观察眼前这一小团材料，如果条件合适，才把它短暂地滚成八音盒风格的高频纹理。

这也正是为什么这一步不能被理解成“看见和弦就一定展开”。程序在这里首先不会问“这里是不是一个和弦”，而更先问“我眼前这团局部材料够不够多，值不值得动”。换句话说，它不是“只要有几颗同拍音就必然滚起来”，而是从一开始就带着明显的保守倾向。这恰恰解释了很多人最直观的不满：明明开了高频装饰，结果还是不够华丽。真正的根源往往不是实现坏掉，而是这套系统本来就不追求“能滚就滚”，它会先审一遍材料是否值得动。

## 它真正吃进去的原料是什么

于是下一步就变成：这一步到底拿什么当原料？答案不是“原始 MIDI 里理想化的一拍和弦”。它吃进去的，是前面几步处理后的局部材料。原始 MIDI 会先被导入，变成内部事件表，可能已经经过 `Transpose / Range / Octave pass`，也可能已经经过 `Trill ornament pass`，然后还会经历一次按时间重排。到了这一层，程序真正看到的才是眼前这一小团可用材料。所以这里的“局部材料”，更接近程序此刻手里真实看到的一小团事件，而不是作曲者脑子里那一拍理论上应该算什么和弦。

这里说的“一团音”，时间尺度其实非常窄。当前主线实现里，这一步看的不是一个宽松的“前后若干拍滑动窗口”，也不是“当前音附近几十个 ticks 都算进来”的松散邻域，它首先看的就是同一个 tick 上的 playable rows。也就是说，对当前实现来说，这团音的第一层时间尺度可以直接写成“宽度等于 0 tick 的同刻事件团”。更容易懂的说法就是：哪些音在同一个 tick 开始，程序就先把它们视为眼前这一团局部材料。所以这里的“团”，本质上不是一段宽时间窗，而是一个同刻 onset cluster。

可即使只看同一个 tick，这团材料也未必简单。因为在走到这一章之前，原始 MIDI 已经先按轨道顺序进入，`Transpose / Range / Octave pass` 可能已经把同拍材料加厚，`Trill ornament pass` 也可能已经插入新的高频事件，后面的时间排序又会把这些事件重新组织到局部 family 里。因此即使现在只看同一个 tick，程序眼里的这团材料也可能已经很厚。前面文档一直说它会受“局部 family”影响，其实并不矛盾：直接喂给这一阶段的，是同一个 tick 上的一团音；而这团音为什么会长成这样，则必须往前追到前面的排序和 ornament 阶段。

## 为什么很多地方根本不会出装饰

很多地方之所以根本不会出装饰，最直接的原因就是这团局部材料不够丰富。当前已确认的规则很硬：如果可用候选少于 `5` 个，程序根本不会进入这一阶段的主要随机判断。用不带内部黑话的方式说，就是眼前这团音太瘦了，不值得滚。这里发生的不是“抽签没抽中”，而是“系统从一开始就觉得这里材料不够，直接跳过”。这正好解释了一类最常见的用户困惑：为什么有些地方明明已经有几颗音了，还是完全没长出八音盒滚动装饰？答案很多时候根本不是随机性，而是材料不够厚。

就算材料已经够厚，程序也不会无条件开始滚。它还会做一次带随机性的继续/放弃判断。如果这次判断决定“放弃”，结果就是这一小段本轮什么都不会长出来。所以“没有装饰”其实至少有两种完全不同的原因：一种是材料本来就不够，系统连尝试都不尝试；另一种是材料够了，但这一次仍然决定不继续。这个区分对读者很重要，因为它意味着并不是所有“没出装饰”都能靠多跑几次解决，有些地方是结构上就不够资格。

## 真长出来时大致会是哪两种样子

当它真的长出来时，当前已经确认的结果至少可以先粗分成两大类。第一类更像一路向上铺到当前时刻。它会先从较低的候选开始，一路往较高的音滚上去，最后一个主音落在当前这一拍。只从听感去想，它更像一串往上爬的细密滚动装饰，最后把重心落回当前时刻。第二类则更像从高处往下压回来。它通常先从较高的候选开始，一路往下滚，但不会把最后一个主音正好钉在当前这一拍上。只听效果，它更像先把高处点亮，再向下压回，并且收得更早一点。所以这两类结果不是简单的“顺序反过来”而已，它们连时间重心都不一样。

## 尾巴是什么，为什么只在一类里出现

其中第一类结果有时还会在主序列结束以后，再补几颗较高的音，形成一小段很短的回落，这就是前面一直说的“小尾巴”。这里的 `post-current` 不是高深术语，它只是说这些补出来的音发生在“当前这一拍之后”。也就是说，主序列已经在当前时刻收束，然后程序又在后面接了一小截短尾巴。这段尾巴不是每次都有，只会出现在前面说的第一类结果里，而且它自己还要再过一次概率判断。

给一个已经闭环的真实例子，会比空讲“高音回落”更容易懂。当前已经确认的一个窗口是 `17580 / current_note = 92 / candidate_count = 16`。如果这个窗口长成“向上铺到当前时刻”的那一类，主序列会是 `52, 52, 52, 56, 56, 59, 59, 59, 59, 64, 64, 68, 80, 80, 80`。如果后面那条小尾巴也出现，那么当前时刻之后还会继续补 `80, 80, 68, 64, 64`。所以这里说的“一小段高音回落”，不是一句空形容，而是真的像主序列先往上堆，到高处以后，再用几颗较高的音往回收一下。

## 为什么这一层特别容易被误读

这一层特别容易被误读，原因也比上一章复杂。上一章至少还有一个很直观的锚点，也就是“这里原本有一个长音”。这一章却不直接依赖一个单一长音，而依赖一小团局部材料、材料是否够厚、这次判断愿不愿意继续、真继续时走哪一种样子，以及第一类结果后面会不会再多一条尾巴。所以如果读者脑中还停留在“看见和弦就自动分解”，那几乎一定会读错这一章。

同样值得说清的是，力度通常解释不了这里的结果。现代 MIDI 用户很容易产生一个直觉：这一段弹得更重，所以应该长出更丰富的高频装饰。当前已确认的实现并不是这样工作的。这一阶段主要依赖的是眼前这团局部材料里到底有什么、材料够不够厚、这次是否愿意继续、长出来时属于哪一种主要样子，以及那条尾巴会不会再出现。它不会主要根据原始力度是不是更大、演奏动态是不是更夸张来决定 ornament 的核心结构。而且新生成出来的 ornament 音，如果没有原始音符可回溯，后面还常常会被写成统一的中等力度。所以这里更准确的说法是：力度会被读取，但它不是这一章的主要驱动变量。

## 这一层在重构里应当保留成什么

如果要现代化重构这一层，最重要的不是先复写内部术语，而是先保住几件真正决定行为的业务事实。首先，它不是一个固定模板展开器；其次，不是每一团材料都有资格进入；再次，就算材料已经够厚，也不保证这一次就一定继续；而真继续时，结果至少还会分成两种明显不同的主要样子，其中一类还可能带一小段发生在当前时刻之后的回落尾巴。只要把这些事实先保住，后面无论你用什么语言、什么模块组织去实现，都还有机会保持原程序的脾气。相反，如果一开始就把这一层改写成“给定一团音，永远按固定顺序全部滚出来”，最后得到的结果固然会更整齐，却也会更不像原程序真正的高频装饰。
