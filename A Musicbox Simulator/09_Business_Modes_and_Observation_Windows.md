# 第九章 业务模式：程序为什么把同一条主链切成几个单独模式

## 这一章要解决的问题

很多人第一次看到这套程序的模式时，会本能地把它们理解成几套彼此并列的功能：`all_off` 是一种处理器，`transpose_only` 是另一种处理器，`trill_only` 又是一种处理器，后面再加上 `mechanize_only` 和 `mech_impossible_only`，看起来就像五个并排摆着的产品开关。这样理解很自然，但不够准确。更接近实际的说法是：程序内部本来有一条连续主链，这些模式只是把同一条主链切成几个更容易单独看的运行方式。

这一章真正要做的事情，不是帮读者背下每个模式的名字，而是把模式背后的业务逻辑讲清楚。只要明白这些模式在观察什么、切掉了什么、保留了什么，读者就不会再把它们误当成五套互不相干的系统，也更容易理解为什么重构时不应该围着这些模式本身打转。

## 先把真正的主链摆出来

到当前为止，这套系统最适合读者理解的主链可以压成这样一条连续过程：先读入 MIDI，建立 tempo、音符、轨道和通道这些原始事实；再把它们变成统一的内部事件表；然后做音区整理与同拍加厚；再进入长音驱动的高频纹理阶段和同刻局部团块驱动的高频滚动装饰阶段；之后再经过机械时间扰动与机械可播放性过滤；最后再分叉导出 `.nts`、processed MIDI、WAV 和动画。

一旦先把这条链看清楚，所谓“模式”就比较容易理解了。它们并不是五种完全不同的音乐生成原理，而更像五种观察方式：有的模式故意只让你看主链前段，有的模式故意把 ornament 中段单独拎出来，有的模式则把机械层单独暴露出来。换句话说，模式的存在首先是为了拆解理解，而不是为了证明系统天生就该被分成这么多块。

## 为什么 `all_off` 不是“什么都不做”

在这些模式里，最容易被误会的就是 `all_off`。很多人看到这个名字，会直接把它翻译成“把所有东西关掉”，仿佛系统几乎不再处理输入，只是原样吐出结果。实际不是这样。`all_off` 更接近一条保守基线：它不会额外强调转调增强，不会额外打开更积极的 ornament 行为，但仍然沿着完整输出链把结果走完。

因此，`all_off`` 不是“绕过系统”，而是“沿着系统走，但尽量少加装饰”。它仍然会经过导入、内部事件构建、后续写出和音频侧的机械可播放性过滤。也正因为如此，`all_off`` 才会保留很多对理解程序很重要的业务后果。最典型的例子就是鼓通道：如果用户拿一首原本带明显鼓点的 MIDI 去跑 `all_off`，最后发现鼓点消失了，正确理解不是“导入器没把鼓读进来”，而是 `all_off` 这条保守链本身主动做了 non-percussion 取舍。

所以 `all_off` 的价值，不在于它代表“系统什么都没做”，而在于它让读者看到：就算不积极追求 ornament，这套系统依然会坚持自己的机械化和非打击乐取向。

## 为什么 `transpose_only` 和 `trill_only` 更像两扇观察窗

`transpose_only` 和 `trill_only` 特别容易让人误会成两套正式产品功能，因为它们都各自产出一份完整结果。可从业务逻辑上看，它们更像两扇观察窗，而且观察的是主链里完全不同的两个部位。

`transpose_only`` 观察的是前段。它最想让你看清的，不是 ornament 好不好，而是原始输入在进入 ornament 之前已经被怎样整理过音区，哪些音被整体移到更合适的工作区间，哪些同拍音又被额外加厚。因此它最适合回答的问题，是“为什么我还没开后面的装饰，材料就已经变厚了”或者“为什么后面的结果看起来不一样，其实变化可能早在 ornament 之前就发生了”。

`trill_only` 则观察的是中段 ornament 链。它把读者的注意力集中到 `Trill ornament pass` 和高频滚动装饰阶段，让人更容易看清某个长音为什么被切细、某个同拍 cluster 为什么长出了高频纹理、某些位置又为什么明明开了装饰却什么都没长出来。它的价值不在于宣称“这就是最终主链”，而在于帮读者单独理解：主链中段那些最容易被听成“风格”的东西，到底是怎样被制造出来的。

这两个模式合在一起，恰好暴露出一个很重要的事实：所谓八音盒听感，并不是某个单独按钮瞬间带来的，而是前段音区整理和中段 ornament 共同叠出来的结果。只盯着其中一个模式，很容易低估另一段对最终听感的影响。

## 为什么 `trill_only` 里面还会继续分叉

如果读者继续往里看，就会发现 `trill_only` 自己还不是一条绝对单纯的路径。旧一点、偏近似的 `trill_only` 路径，和带 stochastic 或 internal rebuild 的 `trill_only` 路径，并不会对所有输入都做完全一样的事。对鼓通道尤其如此：某些路径下，鼓通道可以继续留着；另一些路径下，程序会在 ornament 主链之前先把输入切成 non-percussion baseline。

William Tell 正好是一个很能说明问题的样本。当前主线代码已经明确表明，在 auto baseline 下，stochastic `trill_only` 会先去掉鼓通道，也就是只让 non-percussion 材料进入后续 ornament 主链。所以如果用户观察到 `trill_only` 某些配置下鼓点还在、另一些配置下鼓点没了，这不是文档前后矛盾，而是模式内部本来就还分着不同业务路径。

这一点对重构者特别重要，因为它提醒我们：所谓“模式”不是天然纯净的业务边界，其中有些只是历史上为了观察、回放或兼容某段行为而暴露出来的入口。重构时如果把这些入口直接当成正式系统分层，很容易把过渡期脚手架误认成稳定架构。

## 为什么 `mechanize_only` 和 `mech_impossible_only` 不能被看成“又两种风格”

相比 `transpose_only` 和 `trill_only`，后面这两个模式更容易让人误以为它们也是在改变音乐风格。其实它们切出来看的，是主链更后面的机械层。

`mechanize_only` 单独暴露的是 mechanical timing，也就是时间上的细小机械扰动。它关心的是“什么时候打”，而不是“打什么音”或“多加多少 ornament”。因此它最适合回答的，是为什么结果听起来更像机器、更像真实八音盒的触发，而不是更像精确量化到网格上的 MIDI。

`mech_impossible_only` 则把另一层更容易被忽略的业务单独拎出来，也就是“机械上打不出来的音，音频里还要不要保留”。这条模式最适合解释一种现代用户非常容易困惑的现象：为什么我耳朵里还能听到某个音，但动画或机械表现层却不愿意承认它真的被盘片打出来了。这里被切出来的不是风格，而是音频路径和机械路径之间允许存在的分歧。

所以这两个模式的真正价值，不是让音乐变成另外两种味道，而是提醒读者：主链后段还存在两套和机械真实性有关、但彼此又不相同的判断。

如果只把“鼓通道到底会不会被保留”拿出来单独看，各模式的差别其实很清楚：

| 模式/路径 | 鼓通道怎样处理 | 业务含义 |
| --- | --- | --- |
| `all_off` | 删除 | 保守输出链主动做 non-percussion 取舍 |
| `mechanize_only` | 保留 | 只观察机械时间扰动，不以删鼓为目标 |
| `mech_impossible_only` | 保留 | 观察机械可播放性边界，不先验删鼓 |
| `transpose_only` | 保留 | 观察音区整理，不先切打击乐 |
| `trill_only` + `stochastic_mode = disabled` | 保留 | 旧近似 ornament 路径沿完整 `base grid` 继续走 |
| `trill_only` + `baseline_mode = nonpercussion` | 删除 | 后续 ornament 主链只看 non-percussion 材料 |
| `trill_only` + `baseline_mode = auto`，样本是 `William Tell` | 删除 | `auto` 对该样本会解析成 non-percussion baseline |

## 为什么这些模式会反过来误导读者

这些模式虽然对理解系统有帮助，但它们同时也很容易把人带偏。问题就在于，它们都长得像“独立功能模式”，而用户又天然喜欢把界面上的模式名当成真正的系统边界。这样一来，读者就会不自觉地以为 `transpose_only` 能单独说明整首歌为什么不够华丽，或者 `trill_only` 能单独解释所有机械听感，甚至把 `all_off` 理解成“系统什么都没做”的证据。

但一旦回到主链视角，这些误解就会消失。`transpose_only` 并不能告诉你后面的 ornament 最终长得够不够好，因为它故意停在前段；`trill_only` 也不能独立说明机械时间扰动，因为它把那一层单独切掉了；而 `all_off` 更不是空模式，它只是更保守地沿着同一条主链走下去。换句话说，模式会帮助你拆开系统，但如果你把它们当成系统本身，就会重新把系统看乱。

## 为什么这些模式对重构仍然有用

说到底，这些模式的价值不在于它们应不应该永久保留，而在于它们已经帮我们暴露出主链阶段边界。通过它们，读者能比较清楚地看见：哪一段主要在处理音区，哪一段主要在生成 ornament，哪一段主要在给时间增加机械扰动，哪一段又在判定机械上到底能不能打。对重构来说，这比模式本身是否优雅重要得多。

所以更合理的态度不是把这些模式原封不动当成最终架构中心，而是先承认它们在认知上确实有用，再把它们背后的阶段边界还原回一条正式主链。等正式主链立稳之后，哪些模式还值得继续保留，就应该取决于它们是否还有调试、验证或教学价值，而不是因为历史上界面上正好摆着这几个选项。

这一章最值得记住的一句话是：这些模式真正重要的，不是它们各自像什么功能，而是它们把同一条主链切开给你看。因此模式的价值在于帮助拆解系统，而不是取代系统本身。
