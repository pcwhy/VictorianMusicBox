# 第六章 机械化阶段：`Mechanical Timing Effects` 改的是时间，不是删音

## 这一层为什么最容易和删音混在一起

这是整个系统里最容易和“机械不可播放过滤”混淆的一层。很多人看到“机械化”这个词，会下意识以为它应该是在删掉打不出来的音，或者强行让结果更符合实体盘片限制。这其实把两个不同阶段混在了一起。`Mechanical Timing Effects` 的核心作用不是删音，而是给事件时间加一点小的机械式不稳定感。也就是说，它改的是“什么时候打”，而不是“这个音还能不能存在”。

## 它真正模拟的是什么

理解这一章的第一步，就是把它和“机械不可播放过滤”彻底分开。两者确实服务于同一个大目标，也就是让结果更像一台真实的机械八音盒，但它们碰的是两个完全不同的维度。`Mechanical Timing Effects` 改时间，不直接删音，让结果听起来更像机械而不是完全量化；`Play Mechanically Impossible Notes` 所在的那一层则改可播放性，决定某个音是否还能进入音频输出链，并且它和机械槽位最小间隔有关。如果把这两层写进同一个函数，后面无论调试还是解释，都会越来越乱。

从业务上说，这一步想模拟的是一种很朴素的现象：真实机械乐器不会像数字 DAW 那样，每次都严格落在理论网格点上。即使整体速度稳定，局部触发仍然会带一点点抖动。所以 `Mechanical Timing Effects` 模拟的不是“把节奏打散到认不出来”，而是在原有事件时间附近给出很小的随机偏移。它更像时间拟物化，而不是机械正确性判定。

## 为什么这里总要先讲 division

这里反复出现的 `division` 也不是这一章临时凭空造出来的参数，而是 MIDI 文件头里本来就有的基础时间参数。最容易懂的定义是：`division` 表示一个四分音符里的时间网格被切成多少个 tick。如果继续沿用第一章里最常见、最容易懂的近似，那么在普通读者脑子里，也可以先把它近似读成“一拍大约被切成多少个 tick”。例如当 `division = 96` 时，可以理解成一个四分音符被切成 `96` 个 tick；当 `division = 480` 时，则表示同一个四分音符被切成 `480` 个更细的小格。所以 `division` 越大，时间刻度越细，同样一个节奏位置就可以用更多 tick 去表达。这也解释了为什么前面几章里的很多规则都喜欢写成 `1.4 * division` 或直接写成 `division`：程序很多时间规则不是按固定毫秒写死，而是先相对于这首 MIDI 自己的时间精度来表达。

## 当前实现怎样给事件加轻微抖动

当前主线代码在 `apply_mechanical_timing(...)` 里做的事情其实相当直接。对每个音，它都会计算一个与音高和拍子分辨率有关的扰动窗口：

```text
amplitude = (31.0 / max(1, note.note)) * (division // 8)
```

这条公式在业务上很好理解。`note.note` 越高，分母越大，扰动窗口就越小；`division` 越大，这首 MIDI 的时间刻度越细，扰动窗口也会跟着按比例变化。之所以一定要把 `division` 带进来，是因为如果一首 MIDI 每拍只有很粗的时间格，而另一首 MIDI 每拍切得非常细，那么同样写成“抖动 3 个 tick”，在两首歌里代表的真实时间根本不是一回事。所以程序更合理的做法不是“永远抖同样多的 tick”，而是让抖动幅度随着这首 MIDI 自己的时间精度一起缩放。

后面程序会分别为开始时间和结束时间抽一次小扰动：

```text
start_jitter = round((rng - 0.5) * amplitude)
end_jitter   = round((rng - 0.5) * amplitude)
```

最后再得到新的 `start_tick` 和 `end_tick`。如果算完以后结束时间不再晚于开始时间，程序还会强制把结束时间推到开始时间之后，避免产生无效音。

## 一个最小的数值例子

最小例子可以这样看：假设当前有一个音，音高是 `72`，MIDI 的 `division = 96`。那么先算扰动窗口：

```text
amplitude ≈ (31 / 72) * (96 / 8)
          ≈ 0.43 * 12
          ≈ 5.2 ticks
```

这意味着，对这个音来说，程序这一次给开始时间和结束时间添加的扰动，量级大约只是前后几 `tick`，而不是前后几十 `tick`。再假设原始音符本来是：

```text
start_tick = 384
end_tick   = 480
```

如果这一次随机结果刚好让：

```text
start_jitter = -2
end_jitter   = +3
```

那么新的时间就会变成：

```text
start_tick = 382
end_tick   = 483
```

也就是说，这个音没有被删除，音高也没变。程序只是把它从“绝对整齐地落在网格上”，改成了“稍微早一点开始、稍微晚一点结束”。听觉上它会更像机械系统里带一点点自然抖动的触发，而不是完全数学化的网格播放。

## 为什么它改了时间，却没有删掉音

这也解释了为什么这一层不是删音逻辑。这里发生的事情是：同一个音仍然存在，音高没变，velocity 没变，只是开始和结束时间有了轻微偏移。因此它不是“这个音打不出来，所以删了”，而是“这个音仍然要打，但时间别像完美网格那样太死”。这正是它和 mechanical playability filtering 不能混写的原因：前者保留事件，只改时间；后者则可能直接把某个音从音频路径里裁掉。

既然它改的是事件时间，那么后面所有依赖时间的位置都会受影响，包括 `.nts` 的时间戳、processed MIDI 的 note-on 和 note-off 时间、最终 WAV 里的节奏细部，以及动画里盘片事件的时间分布。但它不会单独决定“这个音是不是机械上允许保留”。所以它影响的是“同一个事件什么时候出现”，不是“这个事件还有没有资格出现”。

## 这一层最容易被误读成什么

这也是为什么听感上它和删音那么容易混。对普通听众来说，时间抖动和删音都会让结果显得“不那么规整”，但它们的来源完全不同：时间抖动会让东西更像机器，删音则会让东西更稀、更受机械约束。如果把两者混成一个按钮，重构出来的系统就很难再解释：为什么有时音没少，却听起来更机械；为什么有时音明显少了，但并不是 timing 在起作用。

真正最容易看错的地方主要有三类。第一类误解，是把“机械化”直接理解成“删掉机械打不出来的音”。这其实把第六章和第七章的职责混在一起了。`Mechanical Timing Effects` 改的是时间位置，让开始和结束不再像电子乐谱那样过于规整；真正决定某个音最后还能不能留在声音路径里的，是后面的 mechanical playability filtering。第二类误解，是以为只要打开这一层，ornament 就会自动变得更多、更华丽。实际上它主要改变的是时间分布，而不是新增 ornament 的数量。换句话说，它会让已有的东西听起来更像机器在工作，却不会替前面的 ornament 阶段凭空创造出更多高频材料。第三类误解，则是把它看成导出前附带跑一下的小后处理，好像不值得被当成正式阶段单独对待。对这套系统来说，这种看法也会把问题看浅，因为它会继续影响后面所有依赖时间的输出链，所以更合理的理解是：它不是一个可有可无的小特效，而是一层正式的时间扰动阶段。

## 这一层和后面的机械过滤怎样分工

看到这里，这一层和后面的机械过滤之间的边界也就清楚了。它负责的是让事件时间听起来更像机械触发，而不是决定哪些音还留在声音路径里。只要把这两件事混在一起，后面无论是调试结果，还是向用户解释“为什么听起来不一样”，都会越来越说不清。
